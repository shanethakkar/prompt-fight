## DECISIONS (running log of nontrivial choices made during implementation)

- 2026-07-03 — **Armor is item-inherent (a property of the worn item), not a separate effect.** The damage pipeline reads `unit.items[*].armor` and folds it into `damage_taken_mult(unit, …)`, so protection is persistent by construction (no ticking effect / no `persistent` flag needed), lives in the inventory, tiers by the item's rating, and prices via the item cost — the cleanest fit for the item system we'd just built. `damage_taken_mult` took an effects list; it now takes the `Unit` to see both effects and worn armor.
- 2026-07-03 — **Items are a structured record layered over the unchanged weapon/tags mechanics.** Rather than re-plumb combat, `Item` objects capture *what* was equipped (for the inspector) while equipping still fans out into `weapon` (re-arm) + `tags` (grant) exactly as before. `kind` is inferred server-side (weapon = has power), so the judge stays as-is and grounding keeps working on `source.items` truthiness — a minimal-blast-radius upgrade (one runtime frontend site + two resolver writes).
- 2026-07-03 — **Aptitude grounding: the server is authoritative on `fit`, defers to the judge only for improvised-vs-unfit.** The judge can't be trusted to reliably grant/deny competence (a live probe showed it calling a wand-equipped stickman "improvised"), and the server *can* verify gear/identity. So `_ground_aptitude` decides `fit` itself (physical → anyone; elemental → summoned creature or a unit with tags/items/an elemental weapon) and only uses the judge's call to split the no-gear case into `improvised` (credible stretch) vs `unfit` (bare over-reach). This mirrors the effectiveness grounding and makes "gear earns the magic" actually hold. Ambition (reach) still throttles a fit specialist's biggest plays, so fit ≠ infallible.
- 2026-07-03 — **Reliability RNG seed lives in `GameState`, minted at the API boundary.** Keeps `resolve_turn` a pure function (replayable/testable): the seed rides in state (set once at `/api/new_match` via `secrets`), and per-roll RNG (P1.1) derives from `(seed, round, active, action_fingerprint)` — same state+action reproduces, fishing-proof + rewrite-cap-bounded. Local hot-seat has no adversarial client; a server-authoritative seed store is deferred to online.

- 2026-07-03 — **P3 entity layer — command mode + bounded-tier effectiveness (user forks).** Building the entity/minion layer (the last piece of the original vision). The user chose **command mode** (no autonomous minion AI — the human commands their units; one command can coordinate several owned units in a priced "combo" like "Harry and Hermione blast the dragon") over auto-fight, for full control; and **bounded-tier** effectiveness (judge picks resisted/neutral/strong/devastating, server owns the multiplier) for the kryptonite-vs-Superman idea. A design-agent review pinned: state → units (`SideState{stickman, entities}`, win = stickman death, decided once at end-of-command with existing precedence); reflect/self-damage returns to the specific source unit; per-target stance consumption; `ActiveEffect.source` stays a Side (an entity dying mustn't cancel dots it applied); stateful judge with **server-minted ids** the judge echoes (never invents) + `normalize` validates every id against the real roster; two-tier stun (stickman → side skips, entity → only its components fizzle); tags grounded as server state from the first summon so P3.3's anti-exploit guardrail is buildable; entities are sandbox-first until a mana pass. Staged P3.0 (units reshape, behavior-identical) → P3.1a (id/roster substrate, still identical) → P3.1b (summon+kits+lifecycle) → P3.1c (combos economy: ≤1 damage **per source_id**, `max_units_per_command`, cap scaling). Deferred: P3.2 items, P3.3 effectiveness/crits, P3.4 charm. Plan in the approved plan file; supersedes DESIGN §7 P3.


- 2026-07-03 — **Effect grammar (Stage A), from the closed 5-category set.** Judge now emits an `Action` = a bundle of 1–3 typed `EffectComponent`s (`damage`/`heal`/`dot`/`hot`/`stat`/`defense`); the server resolves + prices them. Key design pins (from a design-agent stress review): **(cost)** price the whole bundle in aggregate — `ceil((Σ weight)^exp × bundle_mult[n])`, super-additive by count, capped at mana_max — because summing per-component costs was a burst exploit; **(cooldowns)** re-keyed to component *kind*, kept only for heal/defense (the turtle/lock levers), damage/dot/stat ride on mana; **(state)** `PlayerState.effects` is now a **list** (armor/poison/regen/weaken persist + stack: power/speed additive, damage_taken multiplicative, dot/hot coexist, defense single-slot); **(timing)** three-phase turn — start-of-turn over-time ticks with a **new poison-KO path** (a dot can drop you before you act → source wins), staged self-effects (empower+strike boosts the *next* strike), end-of-turn decrement; **(pipeline)** damage = raw → type (vs stance element) → damage_taken (armor, always) → flat stance → floor 0; dot ticks bypass stances/armor (frozen at cast); **(judge)** flat *permissive* component schema (not a discriminated union — Haiku reliability), server normalizes/clamps/caps/drops. Deferred: stun (A.2), resource (A.3), blind/`hit_chance` + reliability + modes (P1), entities (P3). Full plan in the approved plan file; supersedes GAME_MECHANICS §3.
- 2026-07-03 — **Turn model → alternating (Worms-style), from simultaneous** (playtest feedback: clarity). `resolve(state,p1,p2)` → `resolve_turn(state,action)`; simultaneous machinery deleted (speed-order, snapshot-delta, double-KO tiebreak, defense-priority tier); HP floors at 0. Effect/cooldown upkeep = end-of-owner-turn, install-after-tick. **Defenses are persistent stances** (frozen at cast, consume-on-hit, expire next turn). Round cap at round boundary. Events enriched (`target`+`effect`) for client-side result narration; status chips made readable. Judge untouched. Seeded starting-player + probabilistic evade deferred to P1.

Full design in [[DESIGN.md]]; phased rollout in DESIGN §7. Comeback/rubber-band knob deferred. Supersedes SPEC §1 non-goals + GAME_MECHANICS §3/§11.
